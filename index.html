01 · Context背景

Problem and Solution 課題と解決策

The client sells billboard cameras with onboard edge AI that records visitor age, gender, and emotion per detection. All data was stored locally in proprietary DAT files on the device, with no pipeline to surface it for analysis.

クライアントは来客の年齢・性別・感情を検出するエッジAI搭載の看板カメラを販売している。収集されたデータはすべて独自DATファイルとしてデバイスにローカル保存されており、分析に活用する手段がなかった。

A full SaaS platform was designed and built to cover the entire product lifecycle: file ingestion and parsing, a multi-tenant PostgreSQL database, a chart dashboard, PDF and CSV exports, session comparison, Stripe subscription tiers, automated email reports, and an SSE realtime preview for live camera data.

ファイル取込・パースからマルチテナントPostgreSQLデータベース、チャートダッシュボード、PDF/CSVエクスポート、セッション比較、Stripeサブスクリプション、定期メールレポート、SSEリアルタイムプレビューまでを網羅したフルSaaSプラットフォームを設計・構築した。

03 · Data Pipelineデータパイプライン

Parser → Normalizer → Aggregator パーサー → 正規化 → 集計

The camera's DAT format outputs five raw emotion rates per detection. A pipeline of pure functions converts these into three business categories before aggregation. Each stage is independently testable and replaceable.

カメラのDATフォーマットは1検出あたり5種類の感情率を出力する。純粋関数のパイプラインが集計前にこれらを3つのビジネスカテゴリに変換する。各ステージは独立してテスト・置換可能。

lib/normalizers/emotionNormalizer.ts + lib/services/dataProcessor.tsTypeScript
// Camera outputs 5 rates. Dashboard uses 3 business categories.
// surprise → neutral (not a negative signal)
// sad + anger → dissatisfied
export function normalizeEmotion(r: RawPersonRecord) {
  return {
    time:         r.time.substring(0, 2) + ':00',
    happy:        r.happyRate,
    neutral:      r.neutralRate + r.surpriseRate,
    dissatisfied: r.sadRate + r.angerRate,
  };
}

// Orchestrator — called once per API request
export function processRawData(lines: string[]): ProcessedDashboardData {
  const records  = lines.map(parseRawLine);
  const emotions = records.map(normalizeEmotion);
  return {
    kpiStats:        aggregateKPIs(records),
    ageDistribution: aggregateAgeDistribution(records),
    emotionTrends:   aggregateEmotionTrends(emotions),
    timeProfiles:    aggregateTimeProfiles(records),
  };
  // React components receive pre-calculated values — no client-side re-computation.
}

04 · Securityセキュリティ

Auth & Tenant Isolation 認証とテナント分離

Tenant isolation is enforced at the PostgreSQL engine level via Row Level Security — independent of application code. JWTs are stored in httpOnly cookies. Passwords hashed with bcrypt cost factor 12. Login rate-limited to 5 attempts per IP per 15 minutes.

テナント分離はPostgreSQLエンジンレベルのRow Level Securityで強制される。アプリケーションコードとは独立。JWTはhttpOnlyクッキーに保存。パスワードはコスト係数12のbcryptでハッシュ化。ログインはIPあたり15分で5回に制限。

database-schema.sql + app/api/*/route.tsSQL / TypeScript
-- RLS: enforced by the DB engine. Client A cannot read Client B's rows.
CREATE POLICY "Users view own records" ON raw_records FOR SELECT
USING (client_id IN (
  SELECT id FROM clients
  WHERE email = current_setting('app.current_user_email', true)
));

// Every protected API route — JWT verified from httpOnly cookie
let email: string;
try {
  ({ email } = await verifyRequest(
    request.headers.get('authorization'),
    request.cookies.get('auth_token')?.value
  ));
} catch {
  return NextResponse.json({ message: 'Unauthorized' }, { status: 401 });
}
// queryAsUser() sets the session var — RLS handles the rest automatically
const { rows } = await queryAsUser(email, `SELECT * FROM upload_sessions`);

05 · Realtimeリアルタイム

SSE Stream Implementation SSEストリーム実装

Hardware pushes records to an HTTP endpoint which updates an in-memory snapshot store. An SSE stream delivers those snapshots to the browser. No WebSocket server or external pub/sub infrastructure required.

ハードウェアがHTTPエンドポイントにレコードをプッシュし、インメモリスナップショットストアを更新。SSEストリームが接続中のブラウザにスナップショットを配信する。WebSocketサーバーや外部pub/subインフラは不要。

app/api/realtime/stream/route.tsTypeScript
const stream = new ReadableStream({
  start(controller) {
    const enc = (s: string) => new TextEncoder().encode(s);

    // Send latest snapshot immediately on connect
    const init = getSnapshot(billboardId);
    if (init) controller.enqueue(enc(`data: ${JSON.stringify(init)}\n\n`));

    // Poll every 3s — only push when snapshot has changed
    poll = setInterval(() => {
      const snap = getSnapshot(billboardId);
      if (snap && snap.updatedAt !== lastSent) {
        controller.enqueue(enc(`data: ${JSON.stringify(snap)}\n\n`));
        lastSent = snap.updatedAt;
      }
    }, 3000);

    // Heartbeat every 30s keeps connection alive through proxies
    hb = setInterval(() => controller.enqueue(enc(': heartbeat\n\n')), 30000);
  },
  cancel() { clearInterval(poll); clearInterval(hb); },
});
return new Response(stream, {
  headers: { 'Content-Type': 'text/event-stream', 'Cache-Control': 'no-cache' },
});

08 · Design Rationale技術的意思決定

Key Technical Decisions 主要な技術選定の理由

Next.js App Router over a separate API server and SPA

別APIサーバー+SPAではなくNext.js App Routerを選択

A single deployment handles both the React frontend and all API routes — eliminating a second deployment target, CORS configuration, and split environment variables.

単一デプロイでフロントエンドと全APIルートを処理。デプロイ先の分割・CORS設定・環境変数の分離を排除できる。

PostgreSQL RLS over application-layer tenant filtering

アプリレイヤーではなくPostgreSQL RLSでテナント分離

Application-level WHERE clauses fail silently if a developer forgets them. RLS policies are defined once and enforced unconditionally by the database engine — security becomes a database guarantee.

アプリレイヤーのWHERE句は書き忘れると無音で失敗する。RLSポリシーは一度定義すればDBエンジンが無条件に強制するため、セキュリティがDB保証になる。

SSE over WebSockets for realtime

リアルタイムにWebSocketではなくSSEを採用

The use case is strictly server-to-client. SSE is HTTP-native, passes through proxies without configuration, and requires no persistent connection management on the server.

ユースケースは完全にサーバー→クライアントの一方向。SSEはHTTPネイティブでプロキシを設定なしで通過し、サーバー側の永続接続管理が不要。

Server-side aggregation only — no client re-computation

集計はサーバーのみ — クライアントでの再計算なし

Raw records never cross the network. React components receive pre-calculated values. Adding a new chart requires one new aggregator function — no changes to rendering logic.

生レコードはネットワークに流れない。Reactコンポーネントは計算済みの値を受け取るだけ。新チャートの追加は集計関数を1つ追加するだけでよく、レンダリングロジックの変更は不要。
