01 · Context01 · 背景

Problem and Solution課題と解決策

The Problem

課題

The client sells billboard cameras with onboard edge AI that records visitor age, gender, and emotion per detection. All data was stored locally in proprietary DAT files on the device — with no pipeline to surface it for analysis.

クライアントは来客の年齢・性別・感情を検出するエッジAI搭載の看板カメラを販売している。収集されたデータはすべて独自DATファイルとしてデバイスにローカル保存されており、分析に活用する手段がなかった。

The Solution

解決策

A full SaaS platform covering the entire lifecycle: file ingestion and parsing, a multi-tenant PostgreSQL database, a chart dashboard, PDF and CSV exports, session comparison, Stripe subscription tiers, automated email reports, and an SSE realtime preview for live camera data.

ファイル取込・パース、マルチテナントPostgreSQLデータベース、チャートダッシュボード、PDF/CSVエクスポート、セッション比較、Stripeサブスクリプション、定期メールレポート、そしてSSEリアルタイムプレビューまでを網羅したフルSaaSプラットフォーム。

04 · Data Pipeline04 · データパイプライン

Parser → Normalizer → Aggregatorパーサー → 正規化 → 集計

The camera's DAT format outputs five raw emotion rates per detection. The pipeline converts these into three business-meaningful categories before aggregation. Each stage is a pure function with no side effects, making each step independently testable and replaceable.

カメラのDATフォーマットは1検出あたり5種類の感情率を出力する。パイプラインは集計前にこれらを3つのビジネス的カテゴリに変換する。各ステージは副作用のない純粋関数で構成されており、独立してテスト・置換可能。

export function parseRawLine(line: string): RawPersonRecord {
  const fields = line.split(',');
  return {
    date:            fields[0],
    time:            fields[1],
    gender:          fields[4],
    age:             Number(fields[5]),
    detectedEmotion: fields[6],
    neutralRate:     Number(fields[7]),
    happyRate:       Number(fields[8]),
    sadRate:         Number(fields[9]),
    surpriseRate:    Number(fields[10]),
    angerRate:       Number(fields[11]),
  };
}

lib/normalizers/emotionNormalizer.ts TypeScript 
// Camera outputs 5 raw rates. Dashboard uses 3 business categories.
// surprise is grouped with neutral — not a negative signal.
// sad + anger are combined into "dissatisfied".
export function normalizeEmotion(record: RawPersonRecord) {
  return {
    time:         record.time.substring(0, 2) + ':00',
    happy:        record.happyRate,
    neutral:      record.neutralRate + record.surpriseRate,
    dissatisfied: record.sadRate    + record.angerRate,
  };
}

05 · Security05 · セキュリティ

Security Architectureセキュリティ設計

Row Level Security (RLS)

Row Level Security

All six database tables have RLS enabled. Tenant isolation is enforced at the PostgreSQL engine level, independent of application code.

全6テーブルにRLSを有効化。テナント分離はアプリケーションコードとは独立してPostgreSQLエンジンレベルで強制。

httpOnly JWT Cookies

httpOnly JWTクッキー

JWTs are stored in httpOnly cookies. JavaScript cannot read them, which removes token theft via XSS as an attack vector.

JWTはhttpOnlyクッキーに保存。JavaScriptから読み取り不可のため、XSSによるトークン窃取の攻撃経路を排除。

bcrypt Password Hashing

bcryptパスワードハッシュ

Passwords are hashed with bcrypt at cost factor 12. Plain-text passwords are never stored or written to any log.

パスワードはコスト係数12のbcryptでハッシュ化。平文パスワードはシステム内のいかなる場所にも保存・ログ出力されない。

IP-Based Rate Limiting

IPベースレートリミット

Login and password-change endpoints are limited to five attempts per IP per 15 minutes. Implemented without external dependencies.

ログイン・パスワード変更エンドポイントは15分間にIPあたり5回に制限。外部パッケージ不使用のインメモリ実装。

Admin / User Query Separation

管理者/ユーザークエリ分離

queryAsUser() runs with RLS active. queryAsAdmin() bypasses RLS and is restricted to auth lookups and webhook handlers only.

queryAsUser()はRLS有効で実行。queryAsAdmin()はRLSをバイパスし認証ルックアップとWebhookのみに制限。

Private VPC Networking

プライベートVPCネットワーク

In production, RDS public access is disabled. All database traffic is routed privately between App Runner and RDS via a VPC Connector.

本番環境ではRDSのパブリックアクセスを無効化。全DBトラフィックはVPC Connector経由でプライベートルーティング。

06 · Realtime06 · リアルタイム

SSE Stream ImplementationSSEストリーム実装

The realtime path uses Server-Sent Events rather than WebSockets. The data flow is strictly unidirectional: hardware pushes records to an HTTP endpoint which updates an in-memory snapshot store; an SSE stream delivers those snapshots to connected browsers. No persistent connection management or external pub/sub infrastructure is required.

リアルタイムパスはWebSocketではなくServer-Sent Eventsを採用。データフローは完全に一方向で、ハードウェアがHTTPエンドポイントにレコードをプッシュしインメモリスナップショットストアを更新、SSEストリームが接続中のブラウザに配信する。永続接続管理や外部pub/subインフラは不要。

app/api/realtime/stream/route.ts TypeScript 
export async function GET(request: NextRequest) {
  // Auth + plan check omitted — must be 'pro'.
  const billboardId = request.nextUrl.searchParams.get('billboard_id')!;
  let pollTimer: ReturnType<typeof setInterval>;
  let heartbeatTimer: ReturnType<typeof setInterval>;
  let lastSentAt = '';

  const stream = new ReadableStream({
    start(controller) {
      const encode = (s: string) => new TextEncoder().encode(s);

      // Send latest snapshot immediately on connect.
      const initial = getSnapshot(billboardId);
      if (initial) controller.enqueue(encode(`data: ${JSON.stringify(initial)}\n\n`));

      // Poll in-memory store; send only when snapshot has changed.
      pollTimer = setInterval(() => {
        const snap = getSnapshot(billboardId);
        if (snap && snap.updatedAt !== lastSentAt) {
          controller.enqueue(encode(`data: ${JSON.stringify(snap)}\n\n`));
          lastSentAt = snap.updatedAt;
        }
      }, 3000);

      // Heartbeat keeps the connection alive through proxies.
      heartbeatTimer = setInterval(
        () => controller.enqueue(encode(': heartbeat\n\n')),
        30000
      );
    },
    cancel() {
      clearInterval(pollTimer);
      clearInterval(heartbeatTimer);
    },
  });

  return new Response(stream, {
    headers: { 'Content-Type': 'text/event-stream', 'Cache-Control': 'no-cache' },
  });
}

09 · Design Rationale09 · 設計の根拠

Key Technical Decisions主要な技術的意思決定

Next.js App Router over a separate API server and SPA

別APIサーバー+SPAではなくNext.js App Routerを選択

A single Next.js deployment handles both the React frontend and all API routes. This consolidates the deployment target, eliminates CORS configuration, and shares environment variables — a practical trade-off for a project maintained by a small team.

Next.js単体でフロントエンドと全APIルートを処理。デプロイ先が1つになりCORS設定不要、環境変数も共有。小規模チームでの運用を考慮した現実的なトレードオフ。

RLS over application-layer tenant filtering

アプリレイヤーフィルタリングではなくRLSを選択

Application-level filtering fails silently if a developer omits a WHERE clause. PostgreSQL RLS policies are defined once and enforced unconditionally by the database engine — making security a database guarantee rather than a code-discipline requirement.

アプリレイヤーのフィルタリングは開発者がWHERE句を書き忘れると無音で失敗する。PostgreSQL RLSポリシーは一度定義すればDBエンジンが無条件に強制するため、セキュリティがコード規律ではなくDB保証になる。

SSE over WebSockets for the realtime feature

リアルタイム通信にWebSocketではなくSSEを選択

The realtime use case is strictly unidirectional — the server pushes KPI updates to the browser. WebSockets are bidirectional and require persistent connection management that complicates stateless deployments. SSE is HTTP-native and passes through standard proxies without configuration.

リアルタイムのユースケースはサーバー→ブラウザの一方向のみ。WebSocketは双方向でステートレスデプロイを複雑にする。SSEはHTTPネイティブでプロキシを設定なしで通過する。

Server-side aggregation — no client-side re-computation

サーバーサイド集計 — クライアントでの再計算なし

Dashboard data is fully aggregated in the API route before being returned. React components receive pre-calculated values and render only. Raw records never traverse the network, and adding a new chart requires only a new aggregator function — not changes to rendering logic.

ダッシュボードデータはAPIルートで完全に集計してからクライアントに返す。Reactコンポーネントは計算済みの値を受け取ってレンダリングするだけ。生レコードはネットワークに流れない。新チャートの追加は集計関数を1つ追加するだけ。
